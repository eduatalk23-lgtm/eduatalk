## 📋 다음 작업 제안

### 우선순위 1: Margin 클래스 정리 (Spacing-First 정책)

#### 현재 상태
- `components` 디렉토리에서 15개 파일에서 margin 클래스 사용
- 대부분 특수 케이스 (아이콘 정렬, 리스트 들여쓰기, 중앙 정렬 등)

#### 발견된 파일 및 사용 패턴

**1. 허용 가능한 케이스 (유지)**
- `mx-auto`: 중앙 정렬 (허용)
  - `LoadingSkeleton.tsx` (3곳)
  - `EmptyState.tsx` (2곳)
  - `ErrorState.tsx` (1곳)
- `ml-auto`: flex 레이아웃에서 오른쪽 정렬 (허용)
  - `Toast.tsx` (1곳)

**2. 특수 케이스 (검토 필요)**
- `mt-0.5`: 아이콘 정렬 (FormCheckbox.tsx)
  - 체크박스와 라벨의 수직 정렬을 위한 미세 조정
  - **판단**: 특수 케이스로 유지 가능
- `ml-2`: 리스트 들여쓰기 (InstallPrompt.tsx)
  - 순서 있는 리스트의 들여쓰기
  - **판단**: 특수 케이스로 유지 가능
- `mb-1`, `mt-1`: 드롭다운 메뉴 위치 조정 (DropdownMenu.tsx)
  - 드롭다운 메뉴와 트리거 간 간격
  - **판단**: 특수 케이스로 유지 가능
- `my-1`: 구분선 간격 (DropdownMenu.tsx)
  - 메뉴 아이템 간 구분선
  - **판단**: 특수 케이스로 유지 가능

**3. 개선 가능한 케이스**
- 없음 (모두 특수 케이스 또는 허용 가능한 케이스)

#### 결론
현재 `components` 디렉토리의 margin 클래스 사용은 대부분 **특수 케이스**이거나 **허용 가능한 케이스**입니다. Spacing-First 정책 위반이 명확한 케이스는 발견되지 않았습니다.

#### 권장 사항
- 현재 상태 유지
- 새로운 컴포넌트 작성 시 Spacing-First 정책 준수
- 컴포넌트 수정 시 기회가 생기면 점진적 개선

### 우선순위 2: 인라인 스타일 제거

#### 현재 상태
- 일부 컴포넌트에서 동적 스타일을 위해 인라인 스타일 사용
- 대부분 필수 케이스 (동적 width, height 등)

#### 발견된 패턴
1. **ProgressBar 컴포넌트**: 동적 width 값 (필수)
2. **차트 컴포넌트**: 라이브러리 요구사항 (필수)
3. **기타**: 동적 계산 값 (필수)

#### 권장 사항
- 현재 인라인 스타일 사용은 대부분 필수 케이스
- 새로운 컴포넌트 작성 시 CSS 변수 패턴 고려
- 점진적 개선

---

### 우선순위 3: 기타 개선 사항

#### 1. 색상 시스템 표준화
- 하드코딩된 색상 값이 남아있는지 확인
- 디자인 시스템 컬러 사용 확대

#### 2. 접근성 개선
- ARIA 속성 추가
- 키보드 네비게이션 개선
- 스크린 리더 지원 향상

#### 3. 반응형 디자인 개선
- 모바일 레이아웃 최적화
- 브레이크포인트 일관성 확보
