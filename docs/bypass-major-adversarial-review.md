# 우회학과 시스템 (CMS C1.5) — 적대적 SWOT 분석 + 취지 검증

> 작성일: 2026-03-20
> 리뷰어: 30년차 하이엔드 입시 컨설턴트 시점 + 시니어 개발자 시점
> 대상: `학과조회4.accdb` (188MB, 서울 상위 10개 대학) → 웹앱 이관 설계

---

## 1. 시스템 취지 정의

### 1.1 "우회학과"란 무엇인가

학생이 **1지망 학과에 직접 진입이 어려운 경우**, 해당 학과와 **커리큘럼(수업 과정)이 유사한 다른 학과**를 탐색하여:
- 학종(학생부종합전형)에서 **정성평가 역량이 유리하게 작용하는 학과**를 발굴하고
- 학생의 **진단 분석 결과**(교과 이수 적합도, AI 역량 진단, 스토리라인)에 기반하여
- 실질적으로 **합격 가능성이 높은 대안 학과·학교**를 체계적으로 제시하는 시스템.

### 1.2 핵심 가치 명제 (3줄 요약)

1. **커리큘럼 유사도 기반 대안 발굴**: 학과 A의 수업 커리큘럼이 학과 B와 70% 이상 겹치면, A를 포기하지 않고 B로 진입 후 복수전공/전과로 목표 달성 가능
2. **학종 정성평가 역량 매칭**: 학생의 세특·창체·스토리라인에서 드러나는 역량이 해당 학과의 학종 평가 기준과 얼마나 부합하는지 정량화
3. **진단 결과 → 탐색 자동화**: 수동으로 10개 대학 × 수십 학과를 컨설턴트가 하나씩 비교하던 작업을 시스템이 사전 필터링

## 2. 취지 검증: 설계에 핵심 요소가 반영되어 있는가?

### 2.1 컨설턴트 시점 (30년차 하이엔드)

#### 검증 항목 A: "학과의 수업 커리큘럼 비교"

**현재 시스템 상태:**
- `MAJOR_RECOMMENDED_COURSES` (18개 계열 × 일반선택/진로선택) → 고교 교과 매칭만 존재
- `calculateCourseAdequacy()` → 학생 이수 과목 vs 전공 계열 추천 과목 비교
- 대학별 교육과정 PDF 7건 보유 (서울대~건국대)

**적대적 질문:**
> "고교 교과 추천 목록과 **대학 학과의 실제 커리큘럼**은 완전히 다른 차원이다.
> 예: '경영학과'와 '경제학과'는 고교 추천 교과가 거의 동일(미적분, 확률통계, 경제)하지만,
> 대학 커리큘럼은 완전히 다르다(회계/재무 vs 거시/미시경제). 이 차이를 어떻게 반영하나?"

**판정: 부분 반영 (60%)**
- 고교 교과 매칭은 이미 `course-adequacy.ts`로 해결
- **대학 학과 커리큘럼 비교**는 Access DB에 데이터가 있으나 설계 미확정
- 위험: 고교 교과 유사도만으로 "우회" 추천 시, 대학 진학 후 학과 간 괴리 발생

**보완 필요사항:**
- Access DB의 학과별 커리큘럼 데이터 구조 파악 후, **대학 커리큘럼 유사도 매트릭스** 설계
- 유사도 기준: 전공 필수 과목 겹침률, 전공기초 공유률, 복수전공/전과 실적

---

#### 검증 항목 B: "학종 정성평가 역량 고려"

**현재 시스템 상태:**
- 역량 프레임워크: 3영역 × 10항목 (academic/career/community)
- `student_record_competency_scores` → 역량별 A+~C 등급
- `student_record_activity_tags` → 세특/창체 → 역량 연결
- 스토리라인: `student_record_storylines` → 3년 서사 추적

**적대적 질문:**
> "서울대 경영학과 학종과 연세대 경영학과 학종의 평가 기준은 같은 '경영'이어도 완전히 다르다.
> 서울대는 '지적 호기심과 탐구 깊이'를 보고, 연세대는 '리더십과 공동체 기여'를 더 본다.
> 시스템이 '역량 매칭'을 한다면, 대학×학과별 평가 가중치를 어디서 가져오나?"

**판정: 미반영 (30%)**
- 범용 역량 프레임워크(10항목)는 있으나, **대학×학과별 평가 기준 가중치** 데이터 없음
- 현재는 `career_fields` 9종 → 18종 매핑이 전부
- Access DB에 이 데이터가 있는지 미확인 (188MB — 테이블 구조 미파악)

**보완 필요사항:**
- Access DB 테이블 조사로 학과별 평가 기준 데이터 유무 확인
- 없을 경우: 컨설턴트 수동 입력 + AI 보조 생성 체계 필요
- **최소한 "학과가 중시하는 역량 top 3"** 매핑 테이블은 필수

---

#### 검증 항목 C: "진단 분석 결과 기반 학과·학교 탐색"

**현재 시스템 상태:**
- 교과 이수 적합도: `CourseAdequacyResult` (점수 0-100)
- AI 종합 진단: `student_record_diagnosis`
- 배치 판정: `PlacementResult` (Phase 8.5a)
- 졸업생 검색: `AlumniSearch` (Phase 8.6)

**적대적 질문:**
> "진단 결과가 '교육 계열 적합도 85%, 심리 계열 72%'라고 나오면,
> 시스템은 어떻게 '심리학과 대신 교육심리학과'를 추천하나?
> 단순 적합도 점수만으로는 '왜 이 학과인가'를 설명할 수 없다."

**판정: ✅ 구현 완료 (2026-03-20) — 3필터 파이프라인으로 연결**
- 진단 → 적합도 → 학과 추천 파이프라인의 **각 조각은 존재**하지만 **연결이 없음**
- `courseAdequacy` → `targetMajor` → (우회학과 엔진) → 추천 학과 목록 → 배치 가능성
  이 체인이 아직 구현되지 않음

**보완 필요사항:**
- 진단 → 우회학과 연결 파이프라인 명세
- "왜 이 학과인가" 설명 생성 (AI 또는 템플릿)
- 학부모/학생 설득력 있는 근거 자료 자동 생성

---

### 2.2 개발자 시점 (시니어)

#### 검증 항목 D: "Access DB 188MB 파싱 가능성"

**적대적 질문:**
> "탐구DB(46MB)도 mdb-export + CSV → import 파이프라인에 28초 걸렸다.
> 188MB Access DB는 테이블 수, 컬럼 수, 관계 복잡도가 미지수다.
> mdb-export가 188MB를 처리할 수 있나? OLE 오브젝트(이미지/첨부)가 있으면?"

**판정: 고위험**
- 탐구DB: 단일 메인 테이블 + 보조 → 비교적 단순
- 우회학과DB: **10개 대학 × N개 학과 × 커리큘럼** 구조 → 복잡도 미지수
- 188MB는 OLE 오브젝트(이미지, 문서 첨부) 포함 가능성 높음

**보완 필요사항:**
- **사전 조사 필수**: `mdb-tables 학과조회4.accdb` → 테이블 목록
- `mdb-schema 학과조회4.accdb` → 각 테이블 구조
- OLE 컬럼 식별 → 텍스트 데이터만 추출 가능한지 확인
- 탐구DB 파이프라인(`access-parser.ts`, `transformer.ts`) 재사용 가능 범위 판단

## 4. 적대적 리뷰: 컨설턴트 vs 개발자

### 4.1 컨설턴트의 공격 (개발자에게)

**"기술적으로 커리큘럼 유사도를 계산한다고 해서 '우회학과'가 되는 것은 아니다."**

> 현장에서 우회학과를 판단할 때 보는 것:
>
> 1. **학과의 졸업 후 진로 겹침** — 경영학과→컨설팅 vs 경제학과→컨설팅 (O), 경영학과→컨설팅 vs 수학과→컨설팅 (△)
> 2. **대학 내 전과/복수전공 용이성** — 성균관대는 학과간 전과가 비교적 자유, 서울대는 매우 제한적
> 3. **학종 서류에서의 스토리 일관성** — "물리학과 지원인데 문학 동아리만 했다"면 아무리 커리큘럼이 비슷해도 불합격
> 4. **입결(입시 결과) 현실성** — 커리큘럼이 비슷해도 입결이 10점 차이나면 "우회"가 아니라 "하향"
>
> 시스템이 이 4가지를 모두 반영하지 않으면, 컨설턴트는 "참고만 하고 무시"할 것이다.

### 4.2 개발자의 반론 (컨설턴트에게)

**"완벽한 자동화가 아니라, 컨설턴트의 30분짜리 수동 비교를 5분으로 줄이는 도구다."**

> 1. **졸업 후 진로 겹침**: 향후 데이터 추가 가능 (커리어넷, 대학알리미 API)
> 2. **전과/복수전공 용이성**: 정적 참조 데이터로 DB에 넣을 수 있음 (대학별 1행)
> 3. **스토리 일관성**: 이미 `storylines` 테이블이 있고 `career_field`와 연결됨 → 자동 체크 가능
> 4. **입결 현실성**: `university_admissions` 26,309건 + `PlacementResult` 배치 엔진이 이미 있음
>
> **핵심**: 시스템은 "이 학과를 추천한다"가 아니라 "이 학과를 검토해보라"고 제안하는 것이다.
> 최종 판단은 컨설턴트가 한다. 시스템은 **1차 필터링 + 근거 데이터 정리** 역할.

### 4.3 컨설턴트의 재반론

**"그렇다면 '커리큘럼 유사도'보다 '입결 + 학생 역량 매칭'이 더 중요한 1차 필터가 아닌가?"**

> 실제 상담에서 학부모가 묻는 순서:
>
> 1. "우리 아이 성적으로 **갈 수 있는** 학과가 어디인가?" (입결 필터 — 이미 Phase 8.5 배치 엔진 있음)
> 2. "거기 가면 **원래 목표한 것**을 할 수 있는가?" (커리큘럼 유사도 — C1.5 핵심)
> 3. "**생기부가 맞는가?**" (역량 매칭 — 진단 시스템 연동)
>
> 이 순서를 뒤집으면 안 된다. 커리큘럼이 아무리 비슷해도 성적이 안 되면 의미 없다.

### 4.4 개발자의 최종 합의안

**시스템 설계 원칙:**

```
[필터 1] 배치 가능성 (입결) — Phase 8.5 배치 엔진 기사용
    ↓
[필터 2] 커리큘럼 유사도 — C1.5 신규 엔진 (Access DB 기반)
    ↓
[필터 3] 역량 매칭 — 기존 진단 + 스토리라인 연동
    ↓
[출력] 우회학과 추천 목록 (근거 + 점수 + 컨설턴트 코멘트 필드)
```

**UI 원칙:**
- 추천 결과에 반드시 **"왜 이 학과인가"** 설명 포함
- 컨설턴트가 **커스텀 메모를 추가**하여 학부모 공유 가능
- **시스템이 추천하지 않은 학과를 컨설턴트가 수동 추가**할 수 있어야 함
- "추천"이 아닌 **"탐색 후보"**라는 용어 사용 (법적 리스크 완화)

### 최종 권고

**"Access DB 테이블 구조 조사(P0-1) 없이 설계를 확정하면 안 된다."**

188MB DB 안에 무엇이 있는지에 따라 시스템의 범위, 공수, 가치가 완전히 달라진다.
탐구DB(46MB)는 단일 메인 테이블이었지만, 우회학과DB는 **10개 대학 × 학과 × 교과목 × 커리큘럼**의
다차원 관계 데이터일 가능성이 높다.

**착수 순서** (2026-03-20 전체 완료):
```
P0-1~P0-3  ✅ DB 조사 + Import 완료 (3,658학과, 43,316커리큘럼)
P1-1       ✅ 가중 Jaccard (course_type 기반: 전공필수 3.0 ~ 교양 0.5)
P1-2       ✅ 3필터 파이프라인 (커리큘럼 40% + 배치 30% + 역량 30%)
P1-3       ✅ 22개 계열별 10항목 역량 가중치 (constants.ts)
P1-4       ✅ 전과/복수전공 참조 테이블 (10개 대학 시드)
P2         ⏸ 검증 (컨설턴트 블라인드 테스트 — 운영 후 진행)
```
